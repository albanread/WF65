# Golden static baselines

One JSON per corpus file, holding the per-word gate metrics the optimizer must
never regress. **Regenerate with `opt-bench --bless`** (see `../README.md`) and
commit the change in the same PR as the kernel change that moved the numbers.

## These files are CPU-independent

Every metric here is a pure function of the **compiled bytes** — byte lengths
and an iced-x86 instruction decode. The same kernel produces the same codegen on
any x86-64 machine, so these goldens are deterministic and reviewable in a diff,
and a clean checkout reproduces them byte-for-byte.

Two guards prevent comparing numbers that aren't comparable:

- **`kernel_fingerprint`** — BLAKE3 of all `kernel/*.masm` source. If the live
  kernel differs from the one a baseline was blessed against, the gate reports
  "regenerate (--bless)" instead of emitting false regressions. So **any kernel
  edit requires a re-bless** — that is the point: it forces the codegen delta to
  be reviewed.
- **`corpus_file_hash`** — BLAKE3 of the `.f` source. Editing a corpus file
  likewise forces a re-bless of just that file.

## Dynamic numbers live elsewhere

The advisory rdtsc/utime measurements (cycles, microseconds) are CPU- and
machine-dependent and go to git-ignored `*.timing.json` companions — never
committed, never byte-compared, never gating. They only narrate that a static
win shows up in cycles too.

## First bless

Generated by `opt-bench --bless` against the kernel at the recorded
`kernel_fingerprint`. Re-bless after any intentional codegen change.
