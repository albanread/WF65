# WF65 bench spec (harvested from workflow)

## decision_summary
CANONICAL SYNTHESIS of Lenses A (static determinism), B (dynamic realism), C (optimizer coverage). All three converge on the same architecture; conflicts resolved decisively.

CORE DECISION: a STATIC, byte-exact, deterministic gate is the single source of CI truth; the DYNAMIC (rdtsc/utime) layer is advisory-only and structurally incapable of failing CI (zero timing code on the gate's path). Unanimous recommendation of all three lenses and the user's stated bias.

GATE METRICS (hard, fail-CI, deterministic, from debug_words() byte ranges + iced-x86 decode): word_byte_length, call_count_E8, jmp_count_E9, do_lit_count, tail_is_jmp, plus a body_hash (rel32-normalized tripwire) and a top-level kernel_fingerprint guard. ADVISORY (reported, never gate pre-v2): instruction_count, rbp_adjust_count (~0 pre-optimizer because rbp adjusts live inside called primitives; promote rbp_adjust_count to a gate only AFTER the v2 scheduler lands and is re-blessed). All dynamic metrics advisory.

CONFLICTS RESOLVED:
1) Directory: Lens C's bench/ at repo root WINS because the user's schema mandates filenames under bench/corpus/. Baselines in bench/baseline/, checked-in JSON, human-reviewable.
2) Cargo dep: iced-x86 as a [dev-dependency] (gate is a cargo test + bench bin is a test/example target); confirmed NOT currently in Cargo.toml. Pure-Rust, deterministic decoder.
3) Baseline granularity: one JSON file PER corpus file plus a kernel_fingerprint guard so numbers from two kernels are never compared.
4) CALL/JMP counting: ALWAYS via iced-x86 instruction decode, NEVER a raw 0xE8/0xE9 byte scan (those bytes alias inside immediates/.quad literals).
5) do_lit detection: resolve each CALL's absolute target (start+offset+5+rel32) against do_lit's address; do_lit IS in KERNEL_HELPERS and targeted by the kernel's own fold path (compile.masm:168 lea r9,[rip+do_lit]). Fallback: if fragile, gate on call_count+byte_length and demote do_lit_count to advisory.
6) Over-fitting: keep degenerate micro words (fold-chain of identical 1+, dup dup dup, bare +) ONLY as sanity stressors; base verdicts on composable variants (fold-mix with distinct constants, bare-chain, mix-stackops) and the 3 real-world files.

Capture CURRENT (pre-optimizer) codegen as the locked baseline NOW so the first optimizer build yields a real before/after diff. Each transform PR: kernel change -> gate metric improves -> --bless re-commits the golden in the SAME PR.

All facilities VERIFIED in E:\WF65: debug_words()->Vec<(name,start,end)> (src/lib.rs:1849), reset() (:1884), call_xt (:1753), eval (:1785), load_source_file (:1838), rdtsc_word cpuid-serialized (kernel/io.masm:150-155), utime_word QPC (kernel/facility.masm:46-54), fold_*_comp (compile.masm:185/213/242), inline_dup/swap_comp (:753/:791), do_lit (:30, 5B call + 8B .quad = 13B per compile.masm:7-8), compile_comma_no_tco for >r/r>-tailed words (:124), sess()/reset() with RUST_TEST_THREADS=1 (tests/harness.rs:38-52). iced-x86 confirmed ABSENT from Cargo.toml. All corpus vocabulary confirmed in lib/core.f or src/lib.rs PRIMITIVES (square/cube core.f:190-191; constant uses does> @ core.f:78).

## metrics
- **word_byte_length** (static, gate=True, det=True): Per-word compiled body size in bytes = (end - start) from the debug_words() tuple. No decode: pure u64 subtraction. Summed per-file. PRIMARY size axis ('bytes saved'). Reference costs (compile.masm:7-8): call rel32 = 5B; literal = call do_lit + .quad = 13B; folded + = 4B (imm8) / 6B (imm32); inline dup/drop ~8B, inline swap ~11B; TCO leaves length unchanged (0xE8->0xE9 both 5B).
- **call_count_E8** (static, gate=True, det=True): Count of CALL instructions (opcode 0xE8 near rel32, the only call form the compiler emits, compile.masm:96/101) in [start,end) from the iced-x86 decode (Code::Call_rel32_64), NEVER a raw byte scan. Drops on bare-op inline (T2), stack-op inline (T5), literal fold (T1, removes do_lit CALL + operator CALL), TCO (T4). Rising = regression. Verdict metric for bareop and stackinline (where bytes can rise on a real inline win).
- **jmp_count_E9** (static, gate=True, det=True): Count of near JMP instructions (opcode 0xE9 rel32, Code::Jmp_rel32_64) in [start,end) from the iced-x86 decode. The kernel also emits 0xE9 for resolved if/then/loop branches, so this RAW count is reported but the TCO predicate is tail_is_jmp, not raw jmp_count. Short 0xEB jmps counted separately, not gated.
- **tail_is_jmp** (static, gate=True, det=True): Boolean TCO-success predicate: is the body's LAST control-transfer before the terminating 0xC3 ret a 0xE9 JMP targeting another word (not a local if/loop label) rather than a 0xE8 CALL? Immune to non-tail jmp noise. MONOTONE gate: once true it must stay true. Words ending in >r/r> are EXCLUDED from the tailcall category (compile_comma_no_tco, compile.masm:124, never TCO'd) so 'no JMP' is never mistaken for a missed optimization.
- **do_lit_count** (static, gate=True, det=True): Count of `call do_lit` sequences (CALL whose resolved target == do_lit's address, followed by an 8B .quad) in the body. Sharpest literal-fold (T1) signal: a missed fold leaves a 13B do_lit literal + 5B operator CALL. Resolve target = start+offset+5+rel32 vs do_lit's address (do_lit in KERNEL_HELPERS, target of the fold check at compile.masm:168). FALLBACK if fragile: demote to advisory, rely on call_count_E8 + word_byte_length.
- **body_hash** (static, gate=False, det=True): BLAKE3 of the rel32-NORMALIZED body bytes [start,end) (every CALL/JMP/Jcc rel32/rel8 displacement field zeroed before hashing, because displacements encode absolute target-next_ip and differ run-to-run as dictionary layout shifts). 'Did codegen change AT ALL' tripwire that flips on any rewritten opcode/operand even when gate counts coincidentally tie. Drives --bless; CONFIGURABLE gate (default report-only when all gated counts are equal).
- **instruction_count** (static, gate=False, det=True): Total decoded instruction count over [start,end), stopping at the top-level 0xC3 ret. Coarse complement to byte_length; secondary fingerprint of stack-scheduling (T3): coalescing rbp adjusts and keeping TOS in RAX removes spill/reload MOV pairs even when bytes move little. ADVISORY pre-v2 (reported, not gated); surfaces a 'bytes shrink but instr balloons' case.
- **rbp_adjust_count** (static, gate=False, det=True): Count of explicit data-stack-pointer adjustments on RBP (Add/Sub dest RBP + immediate; stk()-style 48 83 C5 ii / 48 83 ED ii) inline in the body. Dedicated T3 fingerprint: the scheduler coalesces N per-op `sub rbp,8` into one `sub rbp,N*8`. ~0 pre-optimizer (adjustments live inside called primitives), so 'must decrease' would be WRONG at baseline. Gate is 'must not exceed baseline', ADVISORY until the v2 scheduler lands, then PROMOTED after a re-bless.
- **file_totals** (static, gate=False, det=True): Per-corpus-file roll-ups (total_bytes, total_instr, total_calls, total_jmps, total_do_lit) over exactly the words that file defines, attributed via the debug_words() before/after delta around the load under one reset() boundary. Headline aggregates ('N bytes saved, M calls eliminated across file X'). Reported, not independently gated (per-word gates cover it).
- **kernel_fingerprint** (static, gate=True, det=True): BLAKE3 of the concatenated kernel/*.masm source bytes the harness compiled, stored at the top of every baseline file (a guard, not a per-word metric). If the live fingerprint differs from the baseline's, the gate reports 'baseline stale for this kernel — regenerate (--bless)' instead of false per-word regressions, preventing comparison across two kernels (the debug_words boundary-semantics drift risk).
- **median_rdtsc_cycles** (dynamic, gate=False, det=False): ADVISORY ONLY, never gates. Median over K>=11 outer reps of (rdtsc delta / M) for an inner loop of M call_xt invocations of the hot word (stack pre-seeded with hot_word_arg; result popped AFTER the loop). rdtsc_word is cpuid-serialized (io.masm:150-155) fencing OoO. Median rejects scheduler/turbo outliers; discard the first warmup rep; reset() between corpus files but NOT between inner reps (keep word JIT-resident, cache-warm); pin thread affinity to one physical core (SetThreadAffinityMask) and raise to HIGH_PRIORITY_CLASS. Reported as median +/- MAD and a current/baseline RATIO. Off the gate path, so it cannot flake CI.
- **min_utime_us** (dynamic, gate=False, det=False): ADVISORY ONLY, never gates. Best-of-N (min, to reject preemption) wall-clock microseconds via utime_word (QPC-backed, facility.masm:46-54) over an outer loop invoking the hot word M times, M large enough that the interval >> QPC resolution. Coarse, frequency-stable cross-check on median_rdtsc_cycles, used only on the longest real-world kernels (dot-prod, mandel-iter) to confirm cycles and wall-clock agree in direction. Reset() between runs. Excluded from the gate (wall-clock, OS-scheduled).

## baseline_format
One checked-in deterministic JSON golden file PER corpus file, at bench/baseline/<basename>.json (e.g. bench/baseline/arith-fold.json). Top-level: { "schema_version": 1, "kernel_fingerprint": "<blake3 of concatenated kernel/*.masm source bytes>", "generated_by": "<git short sha>", "corpus_file_hash": "<blake3 of the .f source so a changed corpus auto-invalidates its golden>", "words": { "<wordname>": { "byte_length": N, "call_count_E8": N, "jmp_count_E9": N, "tail_is_jmp": bool, "do_lit_count": N, "body_hash": "<hex of rel32-normalized bytes>", "instruction_count": N, "rbp_adjust_count": N }, ... }, "file_totals": { "bytes": N, "instr": N, "calls": N, "jmps": N, "do_lit": N } }. Words sorted by name for stable line-by-line diffs. GATE fields (byte_length, call_count_E8, jmp_count_E9, tail_is_jmp, do_lit_count, plus the body_hash tripwire and top-level kernel_fingerprint) are pure functions of the compiled bytes, so the file is byte-stable across runs and reviewable in PRs. instruction_count and rbp_adjust_count are recorded but ADVISORY (carried for the post-v2 promotion). body_hash is over rel32-NORMALIZED bytes (all CALL/JMP/Jcc displacement fields zeroed) so it is position-independent and stable across unrelated dictionary growth. Dynamic numbers go to a SEPARATE git-IGNORED companion bench/baseline/<basename>.timing.json carrying { "<wordname>": { median_rdtsc_cycles, min_rdtsc_cycles, mad_cycles, min_utime_us, reps, warmup, cpu_affinity, priority } } — never committed, never byte-compared, never gating. A bench/baseline/README.md documents the build + CPU used to bless so reviewers know what the advisory numbers mean. JSON (not binary) so an intentional codegen change reads as a clear reviewable delta.

## diff_semantics
Comparison is per-word, keyed by name, between freshly-measured corpus and its committed golden, ONLY when live kernel_fingerprint == baseline.kernel_fingerprint (mismatch short-circuits to 'baseline stale for this kernel — regenerate with --bless', NOT a regression). corpus_file_hash mismatch likewise forces re-bless rather than noise.

GATE (fails CI) on any wrong-direction move of a gate metric: word_byte_length increased, call_count_E8 increased, jmp_count_E9 increased, do_lit_count increased, or tail_is_jmp flipped true->false. Monotone 'never get worse on any static axis'. rbp_adjust_count is gated 'must not exceed baseline' but ADVISORY (report-only) until the v2 scheduler lands, then PROMOTED to a hard gate after a re-bless. instruction_count is reported, not gated.

IMPROVEMENT (any gate metric strictly lower, none higher) is NOT a failure but IS reported and, in default soft mode, flagged 'baseline now understates the optimizer — re-bless' (a soft notice so an improving optimizer does not redden CI). An optional --strict mode makes a stale-better baseline fail, to force refresh.

body_hash CHANGED with all gate counts equal = 'codegen drift': reported, not gated by default (configurable), since a semantics-preserving reschedule that nets identical counts is usually intended.

CORPUS SHAPE: a NEW corpus word absent from the baseline, or a baseline word missing from the corpus, is a hard error until re-blessed — prevents silently dropping coverage.

KNOWN-CORRECT 'bytes up, calls down': transform 2 (bare-op inline) and transform 5 (swap inline) can legitimately INCREASE bytes while DECREASING calls (inline body > 5B CALL). The per-axis report makes this visible; the team must NOT blindly treat any byte increase as failure. Resolution is an intentional --bless. This is why call_count_E8 is the verdict metric for the bareop/stackinline categories, not byte_length.

FALSE-POSITIVE DEFENSES (unanimous): (1) all measurement under session.reset() so dictionary/HERE/LATEST/STATE state is identical; harness asserts depth()==expected and STATE==0 after each load, and asserts re-loading the same .f twice yields identical debug_words byte lengths before trusting any baseline. (2) Only the file's OWN newly-created words attributed, via the debug_words() before/after delta. (3) rel32-normalized body_hash removes address-placement noise. (4) CALL/JMP/do_lit counted from the iced-x86 decoder bounded by [start,end) and stopping at the top-level 0xC3 ret, NEVER a raw byte scan. (5) kernel_fingerprint and corpus_file_hash mismatches short-circuit to 'regenerate'. (6) corpus files are headless pure-compute only (no gfx/canvas/window/event, no . /cr/type in timed paths). HEADLINE numbers: 'calls eliminated' = sum over words of max(0, baseline.calls - live.calls); 'bytes saved' = sum of max(0, baseline.bytes - live.bytes); 'jmps gained' (TCO) = sum of tail_is_jmp false->true transitions; reported per-file and corpus-wide. --bless (or BLESS=1) re-measures and overwrites the goldens; the commit diff is the reviewable record distinguishing WINS from REGRESSIONS. The dynamic diff is advisory: prints median/min cycle and microsecond deltas as median +/- MAD bands and current/baseline ratios, WARNs (never fails) only when non-overlapping bands exceed ~3*MAD, can annotate 'confirmed win' when static and dynamic agree, and has zero authority over the exit code.

## rust_harness_plan
Two consumers, ONE shared measurement core (src/opt_metrics.rs), so the gate and the report cannot drift.

MEASUREMENT CORE (deterministic, no timing) — measure_file(session, path):
1. session.reset() (rolls back to post-bootstrap; src/lib.rs:1884) for isolation.
2. before = session.debug_words() (Vec<(name,start,end)>; src/lib.rs:1849).
3. session.load_source_file(path) (src/lib.rs:1838).
4. assert depth()==expected and STATE==0 (catch an open : or stack imbalance).
5. after = session.debug_words(); the DELTA (new names) is exactly the file's own words — attribute only those.
6. Per new word: byte_length = end - start (pure subtraction). Read body via std::slice::from_raw_parts(start as *const u8, (end-start) as usize), decode with iced_x86::Decoder (bitness=64, ip=start), iterating until the decoder reaches end OR the top-level 0xC3 ret (whichever first, to avoid over-reading padding/the next word). Compute: call_count_E8 (Code::Call_rel32_64), jmp_count_E9 (Code::Jmp_rel32_64), do_lit_count (CALL whose resolved target start+offset+5+rel32 == do_lit_addr resolved once at boot via the compile.masm:168 mechanism, with the 13-byte structural-shape fallback), tail_is_jmp (last control-transfer before the top 0xC3 is Jmp_rel32_64 to another word, not a local label), instruction_count, rbp_adjust_count (Add/Sub dest==RBP + immediate). body_hash = BLAKE3 over a body copy with every CALL/JMP/Jcc rel32/rel8 displacement field zeroed (rel32-normalization).
7. Roll up file_totals; return FileMetrics { kernel_fingerprint, corpus_file_hash, words, file_totals }. Self-check: optionally re-load the same .f in-process and assert identical byte_lengths before trusting the run.

GATE TEST (tests/optimizer_static_gate.rs, #[test], CI-fails): for each corpus file, sess() -> measure_file -> load bench/baseline/<basename>.json -> if kernel_fingerprint or corpus_file_hash mismatch, report 'regenerate' and treat as a soft failure that instructs --bless (do NOT emit per-word regressions). Else per-word assert each gate metric per diff_semantics (byte/call/jmp/do_lit <= baseline; tail_is_jmp monotone true; new/missing word = error). On any wrong-direction move, panic with a self-explaining table: word name, metric, baseline value, live value, the implicated transform (from the corpus mapping), and the iced-x86-disassembled body delta. Zero-flake: single-threaded (RUST_TEST_THREADS=1), no wall-clock, byte-exact inputs.

BASELINE ON DISK + --bless: see baseline_format. opt-bench --bless re-runs measure_file over every corpus file and overwrites the goldens (committed in the same PR as the kernel change so the metric movement is a reviewed diff). opt-bench (no flag) prints the full human report (per-word/per-file bytes saved, calls eliminated, jmps gained) + the advisory dynamic section. --diff <old.json> compares two baselines. --check runs gate logic for non-cargo invocation. --strict makes a stale-better baseline fail.

EXIT-CODE SEMANTICS: gate test passes (exit 0) iff no gate metric regressed AND no corpus-shape change AND fingerprints match; regression or shape change -> panic -> cargo test exit nonzero (CI red). A fingerprint/corpus-hash mismatch is a distinct 'regenerate' outcome (reported loudly; soft failure instructing --bless, not a silent pass). opt-bench --check mirrors this: 0 = clean, 1 = regression, 2 = stale baseline / needs bless. The DYNAMIC layer NEVER touches any exit code — it only prints and WARNs.

## dynamic_layer_plan
Strictly ADVISORY; structurally incapable of failing CI (it lives only in opt-bench's report section and/or a separate #[ignore]-by-default bench entry point — the gate test has ZERO timing on its path, so timer noise cannot affect the CI exit code).

DRIVER — for each corpus file's hot words (the manifest hot_word entries), under a freshly reset() session that has loaded the file:
1. ISOLATION: pin the measuring thread to one physical core via SetThreadAffinityMask (rdtsc deltas on one core's TSC; scheduler cannot migrate mid-measurement) and raise the process to HIGH_PRIORITY_CLASS to reduce preemption. Both via the windows/winapi crate from Rust.
2. SEED: push the hot_word_arg literals onto the data stack (session.push / eval of the literal) so the timed call is pure compute and leaves its result on the stack. Resolve the hot word's xt once (' tick / xt_of) so call_xt has no parser cost.
3. WARMUP: discard the first ~1000 invocations (JIT/page-in, I-cache fill, branch-predictor training).
4. MEASURE cycles: read rdtsc_word (lo hi, already cpuid-serialized, io.masm:150-155) before and after an inner loop of M = manifest 'iterations' call_xt invocations; divide by M; repeat for K>=11 outer reps; take the MEDIAN (robust to the right-skewed scheduler/turbo tail). Also record min_rdtsc_cycles (lower envelope; preemption only ADDS cycles) and MAD (the +/- band). reset() between corpus FILES but NOT between inner reps (keep word JIT-resident, cache-warm). Pop the result AFTER the loop (never in the timed region).
5. CROSS-CHECK time: for the long real-world kernels only (dot-prod, mandel-iter, where total runtime >> QPC resolution), read utime_word (QPC-backed, facility.masm:46-54) around an M-invocation loop and take best-of-N (min) microseconds, to confirm cycles and wall-clock agree in DIRECTION (guards against TSC frequency-scaling artifacts).
6. REPORT: write the git-ignored bench/baseline/<basename>.timing.json (median/min cycles, MAD, min utime us, reps, warmup, affinity, priority). In opt-bench's human report, print each hot word as median +/- MAD band and as a current/baseline RATIO (cycles frequency-invariant for ratios). Express a dynamic 'regression' only as a WARN when non-overlapping bands exceed ~3*MAD; annotate 'confirmed win' when a static instr/byte win and a dynamic cycle drop agree. NOISE FLOOR: only surface a dynamic delta when it exceeds the band (e.g. 3x the MAD of a no-op baseline). Because none of this participates in any assert and the gate test contains no rdtsc/utime/Instant, Windows timer jitter can never flake CI — the dynamic layer narrates 'the byte/instr/call win shows up in cycles and microseconds too', and nothing more.

## open_risks
- rel32 displacements encode (target - next_ip) using ABSOLUTE addresses, so raw body bytes differ run-to-run as dictionary layout shifts. body_hash MUST be over rel32-normalized bytes (displacement fields zeroed via the iced decoder) or it false-positives on every unrelated dictionary change. Single biggest flake risk, designed out by normalization.
- Counting CALL/JMP by scanning raw 0xE8/0xE9 bytes is WRONG (those values appear inside immediates and do_lit's .quad). All counts come from the iced-x86 instruction decode bounded by [start,end) and stopping at the top-level 0xC3 ret.
- do_lit_count depends on resolving a CALL's target to do_lit's address. do_lit is in KERNEL_HELPERS and the kernel itself targets it (compile.masm:168), so it is resolvable — but if exposing it to the harness is fragile, demote do_lit_count to advisory and gate on call_count_E8 + word_byte_length, which capture the same fold win.
- rbp_adjust_count is ~0 pre-optimizer (rbp adjusts live inside called primitives, not in-body); a naive 'must decrease' gate would be wrong at baseline. Gate is 'must not exceed baseline', advisory until the v2 scheduler lands, then promoted after a re-bless.
- tail_is_jmp can be confused by if/then/begin/again branch jmps (kernel emits 0xE9 for those too). Identify the TAIL jmp as the last control-transfer before the top-level 0xC3 whose target is another word (not a local label); report tail_is_jmp separately from raw jmp_count_E9.
- >r/r>-tailed words are compiled via compile_comma_no_tco (compile.masm:124) and intentionally never TCO'd; including one in tco-tail would bake a false 'missing optimization' expectation. They are EXCLUDED from the tailcall category.
- debug_words() ranges may include trailing padding/alignment or a shared EXIT/next tail; if boundary semantics change, all byte/instr counts shift uniformly. Mitigated by locking kernel_fingerprint into each baseline so a kernel change forces an explicit re-bless, and by stopping each decode at its own top-level 0xC3.
- Over-fitting: fold-chain (run of identical 1+), dups (dup dup dup), and bare-add (just +) are degenerate enough the optimizer could special-case them and post inflated wins. Kept as SANITY stressors only; win/loss VERDICTS taken from the composable variants (fold-mix with distinct constants, bare-chain, mix-stackops) and the 3 real-* files.
- Self-modifying fold: try_fold_literal rewinds HERE and rewrites bytes in place, so measuring mid-compilation is meaningless. Only the FINAL post-load word body is measured (the reset/load/snapshot ordering and post-does>/post-TCO timing guarantee this).
- RESET fidelity: the determinism argument rests on reset() restoring a byte-identical post-bootstrap dictionary so corpus words land at stable addresses. Harness asserts re-loading the same .f twice in one process yields identical debug_words byte lengths, and records corpus_file_hash in each golden, before trusting any baseline.
- Headless purity: any accidental gfx/canvas/window/event word or . /cr/type in a timed body would fail headless compilation or pollute timing. Every hot word uses ONLY confirmed core.f/PRIMITIVES vocabulary, leaves its result on the stack, and was verified word-by-word against E:\WF65\lib\core.f and the PRIMITIVES table in E:\WF65\src\lib.rs (arithmetic + - * / mod 2- 1+ 1- 2* 2/; logic and or xor invert lshift rshift; compare = <> < > 0= 0< 0>; stack dup swap drop over rot -rot nip tuck ?dup 2dup 2drop; rstack >r r> r@; loops do ?do loop +loop i j; control if then else begin while repeat until again exit recurse; memory @ ! c@ c! +! cells allot create constant variable buffer:; square/cube in core.f).
- Windows TSC/turbo noise (SMT, frequency scaling, scheduler) is real; if anyone wires rdtsc/utime into an assert it WILL flake. The dynamic layer is kept advisory-only, off the gate path, with affinity-pin + HIGH_PRIORITY_CLASS + median+MAD band-vs-band + current/baseline ratios + utime cross-check on long kernels.
- E:\WF65 and E:\WF64 are near-identical mirrors; the task names WF65 as the target. Author the corpus, baselines, harness, and Cargo.toml change in E:\WF65 to match the cited (verified) line numbers.
